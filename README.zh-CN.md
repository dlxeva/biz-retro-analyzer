# Biz Retro Analyzer / 会后复盘分析 Skill

把杂乱的会后材料，转成有证据支撑的分析资产。

`biz-retro-analyzer` 是一个可复用的智能体 skill，适合分析：

- 会议录音逐字稿
- 会议纪要
- 现场观察记录
- 视频分析文本
- 可选的后续分析对话

并将其转化为：

- 结构化事实底稿
- 项目/事项演进脉络
- 角色诉求分析
- 与会方对议题的理解层级分析
- 关键判断与反向审计
- 下一步动作
- 可复用的方法论笔记

它适用于对话密集、材料混杂、需要持续判断的场景。重点不是只做摘要，而是回答：

- 方向是怎么变化的
- 各方真正想要什么
- 各方到底懂这个议题到什么程度
- 当前哪些问题阻塞推进
- 哪些证据缺口现在可以后置，但后面必须补

---

## 这是什么

这**不是**转录工具，也**不是**通用会议纪要模板。

它是一个**证据优先（evidence-first）的复盘分析 skill**，用于把原始对话材料变成可复用的分析资产。

以下情况尤其适合使用：

- 多场会议并存
- 多种材料混在一起
- 事实与判断混杂
- 多方角色关系复杂
- 试点场景和方向持续变化

---

## 最适合的场景

适合用于：

- 会后复盘
- 合作推进回顾
- 项目演进分析
- 多方协作型复盘
- 现场观察与会议材料联合分析
- “到底发生了什么、什么变了、下一步该怎么办”这类问题

例如：

- 分析一次重要工作会后的多场逐字稿
- 把录音、纪要和现场观察整理成一条清晰脉络
- 审计当前叙事是否真的有证据支撑
- 区分试点场景和系统/产品本体

---

## 不适合的场景

以下场景**不建议**作为主工具使用：

- 普通会议纪要
- 简单 action items 提取
- 情绪分析
- 通用销售电话评分
- 纯转录
- 纯语气风格评估

如果你只需要“摘要 + 下一步动作”，这个 skill 可能过重。

---

## 核心设计

这个 skill 依赖四条原则：

1. **证据优先**  
   先保留原始材料，再做解释。

2. **事实先于判断**  
   先有结构化事实层，再有结论和判断层。

3. **角色真实诉求很重要**  
   区分“他说了什么”和“他真正想要什么”。

4. **理解层级也很重要**  
   不只看“他想要什么”，还看“他真正懂什么、懂到哪一层”。

5. **必要时强制反向审计**  
   如果叙事过于顺滑或自信，就反过来压力测试这些结论。

## 输入模型

这个 skill 支持分层输入。

### 第一层：原始证据

至少提供以下之一：

- 逐字稿
- 会议纪要
- 现场观察记录
- 视频分析文本

如果用户直接给的是音频，应该先转成可用逐字稿，再进入后续分析。

### 第二层：上下文补充（可选）

- 项目/事项背景
- 当前合作关系
- 用户自己的疑问
- 初步设想或假说

### 第三层：后续分析对话（可选）

- 用户与 AI 的分析对话
- 后续复盘讨论
- 中间结论
- 纠偏和修正过程

注意：后续分析对话只能作为**判断层补料**，不能当成原始事实。

---

## 输出模型

根据输入深度和需求，skill 会输出以下一种或多种结果：

1. **结构化事实底稿**
2. **复盘脉络 / 项目线程**
3. **关键判断与反向审计**
4. **当前动作建议**
5. **方法论沉淀**

目标是清晰区分：

- 哪些是已确认事实
- 哪些是推断
- 哪些证据还缺
- 下一步该做什么

### 证据标注

这个 skill 现在支持一套轻量的证据标注约定，用来标识重要判断的证据状态：

- `Status`：`Confirmed` / `Inferred` / `Assumption` / `Needs validation`
- `Strength`：`High` / `Medium` / `Low`
- `Basis`：`Source quote/paraphrase` / `Cross-source pattern` / `User context` / `Later reasoning only`
- `Source anchor`：对高影响判断补充简短来源锚点，如会议、材料、说话人或片段

它尤其适合标注以下内容：

- 角色诉求判断
- 项目边界判断
- 定价或杠杆判断
- 会影响承诺和推进方式的下一步动作

### 高风险判断闸门

当前技能对以下五类判断采用更严格的约束：

- 人物动机
- 客户真实意图
- 预算与报价策略判断
- 组织或政治站位判断
- 合作操盘权与实际控制关系判断

默认规则：

- 除非原始材料表达非常直接且结论保持足够窄，否则不要标成 `Confirmed`
- 如果只来自一次表达或一次会议，最高只写成 `Inferred`，强度为 `Low` 或 `Medium`
- 如果会影响报价、范围、推进节奏或承诺动作，必须同时写出 `Status`、`Strength`、`Basis`、`Source anchor` 和仍待验证事项
- 如果无法给出来源锚点，就降级或不输出

### Participant Understanding Map 默认要求

在 Mode B 和 Mode C 输出中，只要满足以下条件，就默认必须产出 `Participant Understanding Map`：

- 说话人归属可靠
- 参与者或角色之间存在实质差异

只有以下情况可以省略：

- 说话人归属不可靠
- 原始材料过薄，无法支撑人物级理解判断

如果省略，输出中必须显式说明原因。

## 快速开始

### 示例 Prompt

```text
使用 $biz-retro-analyzer 分析以下材料：

1. 三场会议逐字稿
2. 一份现场观察记录
3. 一段后续复盘分析对话

请输出：
- 每份材料的结构化事实底稿
- 一份跨材料脉络
- 角色诉求分析
- 一段反向审计
- 下一步动作清单
```

### 推荐工作流

1. 先只喂原始证据
2. 再补背景
3. 事实层稳定后，再补后续分析对话
4. 如果当前叙事太顺，做一次反向审计
