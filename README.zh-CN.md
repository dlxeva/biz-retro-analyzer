# Biz Retro Analyzer / 复杂对话情报分析 Skill

一个证据优先的复杂对话情报分析 Skill，用来把混乱的项目对话拆成事实记录、推进链、判断审计和下一步行动。

`biz-retro-analyzer` 适合处理复杂协作型项目中的原始对话材料，并把它们转化为可复用的分析资产。

它关注的不只是“大家说了什么”，还包括：

- 哪些事实真的有证据支撑
- 哪些只是参与方自己的说法
- 谁真正理解问题、推进路径、执行约束和决策机制
- 谁提出了某个叙事，谁接住了它，谁让它变成实际行动
- 哪些判断可靠，哪些判断证据不足，哪些还需要验证
- 下一步应该做什么

它适用于对话密集、材料混杂、判断不断变化的复杂协作项目。普通摘要往往太顺滑，也太浅。

---

## 这是什么

这**不是**转录工具，也**不是**通用会议纪要模板。

它是一个**证据优先（evidence-first）的复杂对话情报分析 skill**，用于把原始对话材料变成可复用的分析资产，覆盖参与方理解层级、项目推进结构、影响链、判断风险和下一步行动。

以下情况尤其适合使用：

- 多场会议并存
- 多种材料混在一起
- 事实与判断混杂
- 多方角色关系复杂
- 试点场景和方向持续变化
- 多方协同、谈判或推进摩擦明显

## The Werewolf Test / 狼人杀测试

仓库中包含一个合成评估案例：**The Werewolf Test**。

狼人杀在这里不是游戏功能，而是复杂对话分析的最小试验场。它把真实项目对话里常见的风险压缩到一个小型样本中：

- 公开发言不等于事实
- 说得自信的人可能是错的
- 表达弱的人可能是对的
- 权力和背书可以被传递
- 可疑行为可能只是误判，不一定是恶意
- 最终结果取决于谁把某个叙事推进成了行动

重点不是“盘出狼人”，而是测试 skill 能否在隐藏动机下保持证据纪律。

见：`evaluations/adversarial-dialogue-werewolf/`

## 最适合的场景

适合用于：

- 会后复盘
- 合作推进回顾
- 项目演进分析
- 复杂项目推进结构分析
- 多方协作型复盘
- 现场观察与会议材料联合分析
- “到底发生了什么、什么变了、下一步该怎么办”这类问题

例如：

- 分析一次重要工作会后的多场逐字稿
- 把录音、纪要和现场观察整理成一条清晰脉络
- 审计当前叙事是否真的有证据支撑
- 识别谁提出了框架、谁继承了它、谁改变了下一步路径
- 区分试点场景和系统/产品本体

---

## 不适合的场景

以下场景**不建议**作为主工具使用：

- 普通会议纪要
- 简单 action items 提取
- 情绪分析
- 通用销售电话评分
- 纯转录
- 纯语气风格评估

如果你只需要“摘要 + 下一步动作”，这个 skill 可能过重。

## 核心设计

这个 skill 依赖六条原则：

1. **证据优先**  
   先保留原始材料，再做解释。

2. **事实先于判断**  
   先有结构化事实层，再有结论和判断层。

3. **声称不等于事实**  
   参与方说了什么、暗示了什么、相信什么，都要和材料真正能确认的事实分开。

4. **理解层级很重要**  
   不只看“他想要什么”，还看“他真正懂什么、懂到哪一层”。

5. **影响链很重要**  
   识别一个框架、说法、背书或下一步路径是如何在参与方之间传递，并最终变成行动的。

6. **必要时强制反向审计**  
   如果叙事过于顺滑或自信，就反过来压力测试这些结论。

## 输入模型

这个 skill 支持分层输入。

### 第一层：原始证据

至少提供以下之一：

- 逐字稿
- 会议纪要
- 现场观察记录
- 视频分析文本

如果用户直接给的是音频，应该先转成可用逐字稿，再进入后续分析。

还应尽量识别互动上下文。可用标签包括：

- `formal_meeting`
- `working_session`
- `informal_debrief`
- `private_side_conversation`
- `asymmetric-awareness conversation`

这些上下文很重要，因为非正式或非对称知情材料往往更接近真实顾虑，但也更不稳定。它们适合用于识别顾虑、约束、推进阻力和协同风险，不应在缺少跨材料支持时直接上升为稳定意图、正式立场或承诺。

### 第二层：上下文补充（可选）

- 项目/事项背景
- 当前合作关系
- 用户自己的疑问
- 初步设想或假说

### 第三层：后续分析对话（可选）

- 用户与 AI 的分析对话
- 后续复盘讨论
- 中间结论
- 纠偏和修正过程

注意：后续分析对话只能作为**判断层补料**，不能当成原始事实。

### 非对称知情材料的伦理边界

这个 skill 可以分析非正式或非对称知情条件下的项目材料，但仅限于项目相关的：

- 理解分析
- 协同分析
- 风险分析
- 激励与约束分析
- 决策条件分析

不应用于产出：

- 人格画像
- 超出项目范围的个人利用策略
- 基于他人在非对称知情条件下表达而形成的操控性建议

这类材料导出的判断应保持收窄、内部使用、待验证的默认姿态。

## 输出模型

根据输入深度和需求，skill 会输出以下一种或多种结果：

1. **结构化事实底稿**
2. **复盘脉络 / 项目线程**
3. **推进链 / 影响链**
4. **参与方理解地图**
5. **关键判断与反向审计**
6. **当前动作建议**
7. **方法论沉淀**

目标是清晰区分：

- 哪些是已确认事实
- 哪些是参与方声称
- 哪些是推断
- 哪些证据还缺
- 下一步该做什么

### HTML 输出模式

这个 skill 现在也支持 `HTML` 报告输出模式，适合以下情况：

- 需要生成可直接分享的单页报告
- 需要比 Markdown 更强的视觉分层
- 需要把证据层、项目脉络、反向审计和动作建议放进同一页面

这套 HTML 输出模式由以下文件驱动：

- `references/html-output-mode.md`
- `references/output-schema.json`
- `assets/html-report-template.html`
- `assets/html-report.css`

用户不需要一开始就知道这个模式存在。  
当分析结果已经具备完整结构时，skill 可以在输出阶段主动提示：“这份内容已经适合升级为 HTML 报告页”。

### 证据标注

这个 skill 支持一套轻量的证据标注约定，用来标识重要判断的证据状态：

- `Status`：`Confirmed` / `Inferred` / `Assumption` / `Needs validation`
- `Strength`：`High` / `Medium` / `Low`
- `Basis`：`Source quote/paraphrase` / `Cross-source pattern` / `User context` / `Later reasoning only`
- `Source anchor`：对高影响判断补充简短来源锚点，如会议、材料、说话人或片段

它尤其适合标注以下内容：

- 角色诉求判断
- 项目边界判断
- 价格、筹码或杠杆判断
- 会影响承诺的下一步行动
- 谁控制了决策路径的判断
- 对可疑或矛盾行为的解释
